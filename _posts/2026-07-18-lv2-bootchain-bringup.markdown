QEMU 가상 PCIe 장치(edu/vecadd, [이전 시리즈]({% post_url 2026-07-13-runtime %}))로
드라이버 기초를 다졌으니, 이번엔 실물 하드웨어로 넘어갈 차례다. 라즈베리파이5(SPI 마스터)가
STM32 Nucleo 보드(SPI 슬레이브, FOC 모터 제어 펌웨어가 돌고 있음)에서 실시간 텔레메트리를
커스텀 SPI 캐릭터 드라이버로 받아오는 게 이번 시리즈의 목표다.

## 왜 U-Boot을 굳이 얹었나

라즈베리파이 OS의 기본 부팅 경로는 `펌웨어 → 커널 직행`이라 부트로더가 아예 안 낀다.
근데 BSP/드라이버 채용 공고엔 거의 항상 "부트로더 경험"이 명시돼있어서, 가장 손이 많이
가는 경로 — `펌웨어 → U-Boot(신규로 얹음) → 커널` — 를 일부러 택했다.

## gzip인데 왜 압축 안 풀리는 오류가 나지

U-Boot 콘솔에서 처음 `booti`로 커널을 띄우려 했더니 `kernel_comp_addr_r or
kernel_comp_size is not provided!` 에러. `booti`가 로드된 이미지 첫 2바이트를 압축
매직넘버와 비교하다 매치된 것 같은데, 참고하던 문서엔 "AArch64 커널은 비압축"이라고
적혀있었다. 실제로 메모리를 까봤다:

```
=> md.b ${kernel_addr_r} 0x40
1f8b 0800 ...
```

`1f 8b`는 정석 gzip 매직넘버. **라즈베리파이 OS가 배포하는 `kernel_2712.img`는 실제로
gzip 압축된 상태**였다. 기존(펌웨어 직행) 경로에서 문제가 없었던 이유는 RPi 자체 펌웨어가
내장 압축해제 기능으로 처리해줬기 때문 — U-Boot을 새로 끼워넣으면서 그 기능이 빠진 거다.
`kernel_comp_addr_r`/`kernel_comp_size`(압축해제용 스크래치 버퍼)를 지정해주니 해결.

## 화면이 다 깨진 바이트로 도배되다

압축 문제를 풀고 재시도했더니 이번엔 부팅이 진행되긴 하는데 화면이 완전히 읽을 수 없는
바이트로 뒤덮였다. 커널 패닉 텍스트가 이상하게 보이는 정도가 아니라 진짜 모지바케 —
전형적인 baud rate/클럭 불일치 패턴이다.

원인은 DTB였다. FAT 파티션에 있는 static `bcm2712-rpi-5-b.dtb`를 그대로 `fatload`해서
썼는데, 이건 실측값(정확한 메모리 크기, 예약 영역 등)으로 **fixup되지 않은 원본**이었다.
원래 경로에선 펌웨어가 커널에 넘기기 직전에 항상 이 fixup을 해주는데, U-Boot이 static
파일을 직접 로드하면 그 과정이 빠진다.

U-Boot 소스(`board/raspberrypi/rpi/rpi.c`)를 보니 답이 있었다 — U-Boot이 자신을
실행시킨 펌웨어로부터 **이미 fixup된 DTB 주소를 `${fdt_addr}`에 자동으로 노출**해두고
있었다. `${fdt_addr_r}`(내가 fatload하려던 그 변수)와 이름이 비슷해서 처음엔 헷갈렸는데,
완전히 다른 변수다. DTB를 따로 로드할 필요 없이 이걸 바로 넘기면 됐다.

## 자동 부팅을 만들다가 만난 hush 셸의 함정

수동 부팅은 성공했으니, 이제 `bootcmd`에 시퀀스를 박아넣어 자동화할 차례. 처음엔 이렇게 썼다:

```
setenv bootcmd "...\${kernel_addr_r}..."
```

의도는 "지금 말고 `bootcmd`가 나중에 실행될 때 치환해라"였는데, `printenv`로 확인해보니
백슬래시가 **그대로 저장**돼서 영원히 리터럴 취급되고 있었다. hush 셸에서 `\$`는 "그
자리에서 한 번" 리터럴로 바꾸는 게 아니라, 저장된 문자열 안에 백슬래시 자체가 남아서
나중에 다시 파싱될 때도 계속 리터럴로 남는다. 지연 평가를 의도했으면 애초에 이스케이프를
하면 안 되는 거였다 — 작은따옴표로 감싸니 해결됐다.

## `saveenv`가 통째로 저장한다는 걸 몰라서 생긴, 몇 주 뒤의 재발

여기서 끝난 줄 알았는데, 몇 Phase 뒤(SPI 오버레이를 추가할 때) **완전히 같은 증상**
(`Could not find a valid device tree`)이 다시 나타났다. 이번엔 원인이 살짝 달랐다 —
`set_fdt_addr()`(U-Boot 소스)를 다시 열어보니:

```c
static void set_fdt_addr(void)
{
    if (env_get("fdt_addr"))   // 이미 설정돼있으면 재계산 안 함!
        return;
    ...
}
```

앞서 `bootcmd`를 저장하려고 쳤던 `saveenv`가, **그 순간의 `fdt_addr` 값까지 통째로 같이
저장**해버렸던 거다. 그 이후 매 부팅마다 이 함수는 "이미 있네" 하고 낡은 값을 재사용 —
오버레이로 DTB 크기가 바뀌어 실제 로드 주소가 변해도 반영이 안 됐다. `fdt_addr`를 저장
안 하고 매번 새로 계산되게 고쳤다.

**교훈**: `saveenv`는 방금 바꾼 변수 하나만 저장하는 게 아니라 환경 전체를 저장한다 —
의도치 않은 변수까지 같이 굳어버릴 수 있다.

## DT 매칭 — PCI enumeration과 뭐가 다른가

부트체인을 확보한 뒤엔 가장 단순한 `platform_driver`부터 만들어봤다. Lv1에서 PCI 드라이버가
`pci_device_id[]` 테이블로 매칭됐던 것과 구조는 똑같다 — `of_device_id[]` +
`of_match_table`. 근본적인 차이는 매칭 메커니즘이 아니라 **"디바이스 쪽 존재를 누가
알리는가"**다. PCI는 하드웨어 프로토콜(config space 열거)로 자동 통보되고, platform
디바이스는 그런 하드웨어 프로토콜이 없어서 DT라는 정적 데이터로 사람이 미리 선언해줘야
한다.

빌드된 모듈을 스톡 커널에 `insmod`했더니 `Unknown symbol _dev_info (err -22)` — 크로스
컴파일 트리와 실행 중인 커널의 심볼 버전(CRC)이 안 맞아서였다. 아무리 모듈만 재빌드해도
"빌드 대상"과 "실행 중인 커널"이 다른 이상 해결이 안 되길래, 반대로 **우리가 크로스컴파일한
커널 자체를 실제로 부팅**시켜서 둘을 일치시키는 쪽을 택했다(기존 커널은 안전망으로 그대로
둔 채 별도 파일로 추가).

## SPI 브링업 — ST-Link V3 삽질 3연타

Nucleo 펌웨어를 처음 업로드하려는데 연쇄로 세 가지가 막혔다:

1. PlatformIO가 구식 `hla_swd` transport를 강제 지정 → 이 보드 내장 ST-Link가 V3라서
   최신 OpenOCD의 새 드라이버와 안 맞음. 설정 파일에서 `"swd"`로 직접 고침.
2. `Error: open failed` → WSL2가 USB 디바이스를 아예 못 봄. `usbipd`로 Windows→WSL2
   USB 패스스루 연결.
3. `LIBUSB_ERROR_ACCESS` → 디바이스 노드 권한이 `root:root`. udev 규칙 추가로 해결.

세 개 다 뚫고 나서야 SPI2(Nucleo 슬레이브) ↔ SPI0(RPi5 마스터, `spidev`)로 raw 바이트를
왕복시켰다. `theta` 값이 매 호출마다 바뀌고 `iq_target`이 `iq_meas`에 계속 붙어 따라오는
걸 보고서야, 살아있는 FOC 제어 루프를 실시간으로 읽고 있다는 확신이 들었다.

다음 편은 이 raw SPI 통신을 걷어내고, GPIO 인터럽트로 이벤트 기반 커스텀 캐릭터 드라이버를
만드는 이야기 — 그리고 몇 시간을 날렸던 "코드도 배선도 멀쩡한데 계속 0만 나오는" 미스터리.
