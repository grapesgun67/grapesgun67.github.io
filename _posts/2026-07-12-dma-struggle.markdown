DMA는 이 프로젝트에서 제일 오래 걸린 부분이었다. 순서대로 겪은 문제들을 남긴다.

## 왜 그냥 포인터를 못 넘기나

CPU가 쓰는 커널 가상주소를 장치 레지스터에 그대로 써넣으면 안 된다. 이유 두 가지:

**장치는 프로세스별 페이지테이블을 모른다.** MMU의 가상→물리 변환은 CPU가 `load`/`store`
명령을 실행할 때 CPU 파이프라인 안에서 끝나는 일이다. 장치는 드라이버가 레지스터에
적어준 값 그대로를 버스에 실어보내는 독립적인 "버스 마스터"라, 가상주소라는 개념
자체에 접근할 방법이 없다.

**캐시 일관성 문제.** CPU가 어떤 주소에 값을 썼는데 아직 캐시에만 있고 실제 RAM엔
반영이 안 됐을 수 있다. 장치는 캐시를 거치지 않고 RAM을 직접 읽고 쓰기 때문에, CPU가
쓴 값을 장치가 못 보거나 그 반대의 상황이 생길 수 있다.

`dma_alloc_coherent()`가 이 문제를 해결해준다 — 캐시 일관성이 보장되는 메모리를 잡아주고,
**같은 물리 메모리를 가리키는 두 개의 주소**를 돌려준다: CPU가 쓸 가상주소, 그리고
장치가 쓸 버스주소.

```c
void *virt = dma_alloc_coherent(&pdev->dev, size, &dma_handle, GFP_KERNEL);
```

## 28비트 DMA 마스크 — 왜 이런 제약이 있나

`dma_alloc_coherent`를 부르기 전에 `dma_set_mask_and_coherent(&pdev->dev,
DMA_BIT_MASK(28))`를 먼저 불러야 한다. edu는 기본적으로 256MiB(28비트)까지만 가리킬 수
있게 설계돼 있다.

이건 edu만의 특이한 제약이 아니라, 실제 하드웨어에서도 흔한 패턴이다. DMA 주소
레지스터가 몇 비트짜리로 만들어졌느냐는 칩 설계 당시의 비용 문제였다 — 넓게 만들수록
실리콘 면적이 더 든다. ISA DMA는 24비트, 32비트 시대의 수많은 PCI 장치는 32비트까지만
지원했다. 마스크 설정을 빼먹으면, 커널이 장치가 못 가리키는 범위에 메모리를 잡아버릴
수 있고, 그 주소를 좁은 레지스터에 욱여넣으면 상위 비트가 잘려서 완전히 엉뚱한 곳을
가리키게 된다 — 크래시가 아니라 조용한 데이터 오염으로 나타난다.

## 비동기라는 걸 몰라서 생긴 첫 삽질

DMA 왕복 테스트를 처음 짤 때, 전송 시작 레지스터를 쓰자마자 바로 결과를 비교했다.
당연히 틀렸다. QEMU 소스를 보니 이유가 있었다:

```c
if (timer) {
    timer_mod(&edu->dma_timer, qemu_clock_get_ms(QEMU_CLOCK_VIRTUAL) + 100);
}
```

DMA 시작 명령은 **즉시 처리되는 게 아니라 100ms 뒤로 예약**된다(실제 DMA 엔진이 시간이
걸리는 걸 흉내내려는 의도로 보인다). CPU가 시작 명령을 내리고 바로 결과를 확인하면,
아직 아무 일도 안 일어난 상태를 보는 것이다. edu 스펙 문서에 있던 대로
`while (cmd_register & 1);` 폴링 루프를 넣어야 했다.

## 에러 없이 실패하는 버그 — 버스마스터링

폴링까지 넣고 나서도 왕복시킨 데이터가 여전히 원본과 달랐다(정확히는 목적지 버퍼가
처음 그대로 0). `pci_enable_device()`는 불렀는데 결과가 이상했다.

QEMU 소스의 `pci_dma_read`/`pci_dma_write`가 `bus_master_enable_region`이라는 걸
거친다는 걸 발견했다. 그리고 훨씬 전, Phase 0에서 그냥 `lspci -vvv`로 처음 확인했을 때
이미 힌트가 있었다:

```
Control: I/O+ Mem+ BusMaster- SpecCycle- ...
```

`pci_enable_device()`는 `Mem+`(메모리 공간 디코딩)만 켜준다. **`BusMaster` 비트는
별도로 켜야 한다** — `pci_set_master(pdev)`. 이게 꺼져 있으면 장치가 스스로 버스에
나가서 RAM을 읽고 쓸 권한이 아예 없다. 그리고 결정적으로, edu.c는 `pci_dma_read`/
`write`의 성공/실패 여부를 체크하지 않는다 — 그래서 이 실패가 **에러 메시지 하나 없이**
조용히 무시됐다.

```c
pci_enable_device(pdev);
pci_set_master(pdev);   // 이거 빼먹으면 DMA가 조용히 no-op이 된다
```

가장 크게 남은 교훈: **"에러가 안 났다"가 "제대로 동작했다"를 보장하지 않는다.**
완료 신호(폴링이 끝났다, 인터럽트가 왔다)만 믿고 넘어갔으면 이 버그를 영영 못 잡았을
것이다 — 실제 바이트 값을 하나하나 찍어서 비교하는 습관이 결정적이었다.

ioctl + mmap으로 유저스페이스까지 연결하는 이야기.
